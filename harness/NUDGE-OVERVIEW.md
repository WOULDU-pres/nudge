# NUDGE — 프로젝트 원리 및 기술 해설

---

## 1. 이 프로젝트가 해결하는 문제

최고의 영업사원은 수천 번 거절당하며 감각을 만든다.
어떤 첫마디가 먹히는지, 어떤 타이밍에 가격을 꺼내야 하는지 —
10년 넘게 몸으로 부딪혀야 쌓이는 노하우다.

문제는 세 가지다:

```
1. 느리다
   전략 하나를 검증하려면 실제 고객 수백 명에게 시도해야 한다.
   10가지 전략을 비교하려면 몇 달이 걸린다.

2. 비가역적이다
   실패한 전략에 노출된 고객은 돌아오지 않는다.
   실험 비용이 곧 고객 이탈이다.

3. 전수가 안 된다
   베테랑의 노하우는 그 사람 머릿속에만 있다.
   "느낌"으로 하는 영업은 매뉴얼로 만들 수 없다.
```

NUDGE는 이 세 가지를 동시에 해결한다.
실제 고객 대신 **200명의 가상 고객**에게 전략을 테스트하고,
AI가 결과를 평가해서 **전략을 스스로 진화**시킨다.
사람이 10년 걸려 쌓는 멘탈맵을, 하루 만에 시뮬레이션으로 압축한다.

## 2. 전체 작동 원리 — 한 장으로 이해하기

```
┌───────────────────────────────────────────────────────────────┐
│                                                               │
│   [ strategy_prompt.md ]                                      │
│         │                                                     │
│         ▼                                                     │
│   ┌─── RALPH 내부 루프 (1회 시뮬레이션) ──────────────────┐   │
│   │                                                       │   │
│   │  H  전략 생성 ─── AI가 설득 전략 3개를 만듦           │   │
│   │  P  대상 선택 ─── 200명 중 테스트 대상 선택           │   │
│   │  A  대화 실행 ─── 전략 × 페르소나 조합으로 대화       │   │
│   │  E  평가 채점 ─── AI Judge가 4차원 + 퍼널 채점        │   │
│   │  R  패턴 분석 ─── 잘 된 대화 vs 못 된 대화 비교       │   │
│   │  L  학습 추출 ─── 다음에 써먹을 포인트 정리           │   │
│   │                                                       │   │
│   │  → 결과: avg_score, cluster_coverage                  │   │
│   └───────────────────────────────────────────────────────┘   │
│         │                                                     │
│         ▼                                                     │
│   점수가 올랐으면 → keep (이 방향 유지)                       │
│   점수가 안 올랐으면 → discard (되돌리기)                     │
│         │                                                     │
│         ▼                                                     │
│   strategy_prompt.md 수정 → 다시 RALPH 실행                   │
│   (이 바깥 루프가 autoresearch — 무한 반복)                   │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

핵심 구조는 **2겹 루프**다:

- **안쪽 루프 (RALPH)**: "이 전략이 먹히나?" → 대화 시뮬레이션 + 평가
- **바깥 루프 (autoresearch)**: "전략을 만드는 프롬프트 자체를 어떻게 바꾸면 더 좋은 전략이 나오나?"

즉, 전략을 진화시키는 게 아니라 **전략을 만드는 방법론을 진화**시킨다.

## 3. 전략 생성 방식 — AIDA 퍼널 기반

### 왜 AIDA인가

고객이 물건을 사기까지는 단계가 있다:

```
Attention  →  Interest  →  Desire  →  Action
 "뭐지?"     "좀 더       "이거       "살게요"
              알아볼까?"    갖고싶다"
```

이걸 AIDA 퍼널이라 부른다 (1898년, E. St. Elmo Lewis).
NUDGE의 모든 전략은 이 4단계 구조로 설계된다.

### 전략 하나의 생김새

AI가 생성하는 전략은 이런 구조다:

```json
{
  "strategy_id": "strategy-daily-cost-empathy",
  "hypothesis": "하루 비용 프레이밍 + 공감 오프닝이 가성비 민감 고객에게 효과적",

  "funnel": {
    "attention": {
      "hook_type": "공감형",
      "opening_line_guide": "바쁜 일상에서 건강 챙기기 어려우시죠",
      "target_emotion": "공감"
    },
    "interest": {
      "value_framing": "하루 500원으로 건강관리 가능",
      "information_depth": "비교데이터",
      "engagement_trigger": "지금 드시는 영양제랑 비교해보실래요?"
    },
    "desire": {
      "emotional_driver": "건강불안 해소",
      "proof_type": "ROI계산",
      "objection_preempt": "30일 환불보장으로 리스크 제거"
    },
    "action": {
      "cta_style": "시험제안",
      "urgency_type": "없음",
      "fallback": "관심 있으시면 성분표만 보내드릴까요?"
    }
  },

  "tone": "차분하고 데이터 중심",
  "persona_adaptation": "가성비 민감 고객에겐 숫자 먼저, 건강불안 고객에겐 증상 먼저"
}
```

각 단계에서 **정확히 뭘 해야 하는지**가 명시된다.
"좋은 점을 설명한다" 같은 모호한 지시가 아니라,
"턴 0에서 공감형 hook → 턴 1에서 하루 500원 ROI 비교 → 턴 2에서 시험 제안"처럼
**턴별 목표가 정해진 대화 설계도**다.

### 턴과 퍼널의 매핑

```
턴 0  세일즈: 첫 인사 (Attention hook)
      고객:   첫 반응                     ← Attention → Interest 전환점
턴 1  세일즈: 가치 설명 + 증거 제시
      고객:   추가 반응                   ← Interest → Desire 전환점
턴 2  세일즈: 행동 유도 (CTA)
      고객:   최종 반응                   ← Desire → Action 전환점
```

3턴(6메시지)이라는 짧은 대화에서 퍼널 전체를 커버해야 하므로,
**매 턴이 다음 단계로 올라가는 계단 구조**가 된다.

## 4. 가상 고객 페르소나 — 200명의 설계

### 페르소나가 왜 200명인가

실제 고객층은 다양하다.
가성비만 따지는 사람, 건강불안이 큰 사람, 세일즈 자체를 혐오하는 사람.
한 가지 전략으로 모두를 설득할 수 없다.

NUDGE는 200명의 페르소나를 **심리학 기반 클러스터 8개**로 분류한다:

```
budget_practical    ← 가성비 최우선, 숫자로만 움직임
health_anxious      ← 건강 걱정이 크지만 정보 과부하 상태
hard_sell_resistant ← 세일즈 화법 자체에 강한 거부감
impulse_curious     ← 호기심은 있지만 결정이 빠르지 않음
evidence_driven     ← 성분, 논문, 인증 등 근거를 요구
social_follower     ← 주변 사람들이 하는 걸 따라감
routine_seeker      ← 익숙한 걸 바꾸기 싫어함
low_attention       ← 긴 설명을 안 듣고, 3초 안에 판단
```

### 페르소나 하나의 구조

각 페르소나는 `profile.json`(구조화 데이터)과 `soul.md`(자연어 캐릭터 시트)로 구성된다.

예시 — P001 김도윤 (20대, 대학생 겸 알바생):

```
soul.md에 담긴 것:
  • 나는 누구인가: 서울에서 학교 다니면서 편의점 야간 알바를 하는 20대.
    계산기 두드려도 납득되는 제품만 남긴다.
  • 대화 스타일: "그래서 하루 얼마인데요?"가 자동으로 나옴.
    길게 설명하면 중간에 끊고 숫자부터 확인.
  • 설득 포인트: 1일 섭취 비용이 먼저 보이면 잘 듣는다.
  • 절대 안 먹히는 것: 가격을 흐리고 감성만 강조하는 방식.
  • 예상 반론: "편의점 음료로도 버티는데 굳이?" / "학생이라 부담돼요"
  • 관심 신호: 하루 비용을 스스로 다시 계산하기 시작함
  • 이탈 신호: "좋긴 한데 가성비가 애매하네요"
```

**핵심 설계**: 고객 에이전트(LLM)에는 "설득 포인트"와 "관심 신호"를 전달하지 않는다.
이걸 알면 스스로 "나는 이렇게 하면 설득된다"를 알고 연기하게 되기 때문이다.
에이전트는 성격과 상황만 알고, 자연스럽게 반응한다.

## 5. 평가 방식 — AI Judge의 4차원 + 퍼널 채점

### 4차원 평가 (각 0-25점, 총 0-100점)

```
┌─────────────────┬───────────────────────────────────────────────┐
│ engagement      │ 고객이 대화에 관심을 보였는가?                │
│ (참여도) 0-25   │ 질문을 했는가? 단답만 했는가?                 │
│                 │ 21+: 자발적 질문, 대화 주도                   │
│                 │  0-5: 무반응, 즉시 이탈                       │
├─────────────────┼───────────────────────────────────────────────┤
│ relevance       │ 에이전트가 고객 유형에 맞게 대응했는가?       │
│ (적합성) 0-25   │ 고객의 상황에 연결된 접근이었는가?            │
│                 │ 21+: "내 이야기를 한다"고 느낌                │
│                 │  0-5: 완전히 엇나간 접근                      │
├─────────────────┼───────────────────────────────────────────────┤
│ persuasion      │ 반론을 효과적으로 다뤘는가?                   │
│ (설득력) 0-25   │ 논리적 근거와 프레이밍이 효과적이었는가?      │
│                 │ 21+: 반론을 재프레이밍해서 고객 납득           │
│                 │  0-5: 반론 무시 또는 압박, 역효과              │
├─────────────────┼───────────────────────────────────────────────┤
│ purchase_intent │ 대화 끝에 구매/관심 방향으로 기울었는가?       │
│ (구매의향) 0-25 │ 구체적 다음 행동을 언급했는가?                │
│                 │ 21+: "주문할게요", "링크 보내줘요"             │
│                 │  0-5: "다시 연락하지 마세요"                   │
└─────────────────┴───────────────────────────────────────────────┘
```

이 4차원은 AIDA 퍼널과 자연스럽게 매핑된다:

```
engagement      ←→  Attention → Interest (관심 끌기 성공 여부)
relevance       ←→  Interest 단계 (고객 pain point에 연결했는가)
persuasion      ←→  Interest → Desire (반론 해소로 욕구 강화했는가)
purchase_intent ←→  Desire → Action (구매 행동까지 갔는가)
```

### funnel_progress 보조 지표 (0-3)

4차원 점수와 별도로, **고객이 퍼널에서 어디까지 도달했는지**를 추적한다:

```
0: Attention 이탈  ← 첫 반응에서 관심 없음. 대화 확장 실패.
1: Interest 도달   ← 질문/정보 요청은 했으나, "갖고 싶다"까지 안 감.
2: Desire 도달     ← "괜찮네요", "얼마예요?" 표현. 구매 행동은 미도달.
3: Action 도달     ← 구매 의사 또는 "링크 보내줘요" 등 구체적 행동.
```

이 지표가 중요한 이유: "어디서 이탈하는가"를 정밀 진단할 수 있다.

예: "이 전략은 Attention→Interest 전환율이 80%지만,
     Desire→Action 전환율이 10%다"
→ 다음 세대는 **Action 단계만 집중 개선**하면 된다.

### 한국어 문화적 맥락

Judge는 한국어 대화의 뉘앙스를 이해한다:
- "생각해볼게요" = 대부분 거절 (사교적 거절)
- "괜찮아요" = 거절
- "좀 더 알아볼게요" = 약한 관심이지만 행동은 안 함

이런 맥락 없이 채점하면 점수가 부풀려진다.

## 6. 학습 누적 — Strategy Ledger (전략 원장)

### 문제: 세대 간 망각

RALPH 루프가 세대를 거듭하면, 이전 세대의 학습이 사라진다.
세대 10이 세대 9의 결과만 보고, 세대 1~8은 잊어버린다면?
→ 이미 실패한 전략을 다시 시도하게 된다.

### 해결: Strategy Ledger

모든 세대의 전략 시도와 결과를 **압축된 원장**으로 누적한다.

```
strategy_ledger.json

generations:  ← 세대별 기록
  세대 1: 전략 3개 시도, avg 55.7, ROI 전략이 budget_practical에서 효과
  세대 2: 전략 3개 시도, avg 61.2, 공감형 오프닝이 전체적으로 상승
  ...
  세대 10: 전략 3개 시도, avg 72.4, ...

cumulative_insights:  ← 전 세대 누적 인사이트
  never_repeat:      ← 확정된 실패 (3세대 연속 실패한 접근)
    - "hard_sell_resistant 클러스터에 고압적 톤 사용 금지"
    - "충격형 hook + 직접제안 CTA 조합은 전 클러스터에서 실패"

  proven_effective:   ← 확정된 성공 (3세대 연속 성공한 접근)
    - "budget_practical에 하루 비용 프레이밍 — 일관 60+"
    - "소프트CTA + 시험제안이 resistant 클러스터에서 안정적"

  best_score_ever: 78.5  ← 역대 최고 기록
```

### context 폭발 방지: 3-레이어 압축

10세대 분량의 전체 대화 원문을 LLM에 넣을 수 없다.
그래서 3단계로 압축한다:

```
Layer 1: cumulative_insights (항상 전달, ~500 토큰)
  → never_repeat + proven_effective + best_score_ever
  → 세대가 아무리 늘어도 크기 고정

Layer 2: 최근 3세대 상세 (~1,500 토큰)
  → 가장 최근 실험 3개의 전략/점수/학습

Layer 3: 과거 세대 1줄 요약 (~50 토큰/세대)
  → "세대 1: avg 55.7, ROI 전략, budget_practical만 효과"

총 context (10세대 기준): ~2,350 토큰 ← 관리 가능
```

## 7. 하네스 구조 — "코드가 아니라 설계도"

### 하네스란?

NUDGE 하네스는 **코드가 아니다**.
AI 에이전트(Claude Code, Codex 등)에게 주는 **설계도**다.
에이전트가 이 폴더를 읽으면, 스스로 코드를 생성하고, 실행하고, 진화시킨다.

```
ralphthon-harness/       ← 이 폴더가 "하네스" (설계도)
├── MASTER-PROMPT.md     ← 에이전트 진입점 (이것만 읽으면 시작)
├── PLAN.md              ← 설계 계획서
│
├── architecture/        ← 시스템 설계 문서
│   ├── system-overview.md    ← 전체 그림 (2-레벨 루프)
│   ├── execution-phases.md   ← 3단계 실행 흐름
│   ├── ralph-loop.md         ← RALPH 내부 루프 상세
│   ├── autoresearch-loop.md  ← 외부 루프 상세
│   └── tech-stack.md         ← 기술 스택 + 제약
│
├── contracts/schemas/   ← 데이터 구조 계약
│   ├── strategy.schema.json          ← 전략 스키마 (AIDA 퍼널)
│   ├── strategy-ledger.schema.json   ← 전략 원장 스키마
│   ├── conversation-session.schema.json
│   ├── evaluation-result.schema.json
│   └── ...
│
├── prompts/             ← 시스템 프롬프트 템플릿
│   ├── strategy_prompt.md        ← 전략 생성 프롬프트 (진화 대상!)
│   ├── sales-agent-system.md     ← 세일즈 에이전트 프롬프트
│   ├── customer-agent-system.md  ← 고객 에이전트 프롬프트
│   ├── judge-system.md           ← 평가 Judge 프롬프트
│   ├── hypothesize-system.md     ← 전략 생성 보조 프롬프트
│   ├── reason-system.md          ← 패턴 분석 프롬프트
│   └── learn-system.md           ← 학습 추출 프롬프트
│
├── criteria/            ← 품질 기준
│   ├── evaluation-rubric.md      ← 4차원 평가 루브릭
│   ├── quality-gates.md          ← 단계별 통과 기준
│   └── autoresearch-rules.md     ← keep/discard 판단 기준
│
├── data/personas/       ← 200명 페르소나 (고정, 수정 금지)
│   ├── P001/ (profile.json + soul.md)
│   ├── P002/
│   └── ... P200/
│
└── program.md           ← autoresearch 실험 지시서
```

### 에이전트가 하는 일 (3단계)

```
Phase 1: 코드 생성
  에이전트가 architecture/, contracts/, prompts/ 를 읽고
  Python 코드를 자동 생성한다.
  → output/ 폴더에 RALPH Loop, 대화엔진, Judge, LLM 클라이언트 등

Phase 2: 베이스라인 실행
  생성한 코드로 첫 시뮬레이션을 돌려서 기준 점수를 잡는다.
  → results.tsv에 baseline 기록

Phase 3: autoresearch 루프 (NEVER STOP)
  strategy_prompt.md를 수정 → 시뮬레이션 → 점수 비교 → keep/discard
  → 사람이 중단할 때까지 무한 반복
```

**핵심 규칙**: 에이전트는 사람에게 묻지 않는다. 코드가 깨지면 스스로 고친다.
strategy_prompt.md만 수정한다. 평가 기준은 절대 바꾸지 않는다.

### 수정 가능 / 수정 불가

```
수정 가능 (autoresearch가 실험마다 바꾸는 것):
  └── strategy_prompt.md ← 이것 하나만

수정 불가 (고정된 것):
  ├── 코드 (Phase 1 이후 고정)
  ├── 페르소나 데이터 (200명)
  ├── Judge 프롬프트 / 평가 루브릭
  ├── 고객 에이전트 프롬프트
  └── results.tsv 형식
```

평가 기준을 바꾸면 과거 실험과 비교가 불가능해진다.
진화 알고리즘에서 **fitness function을 고정**하는 것과 같은 원리다.

## 8. 대화 실행의 상세 흐름

하나의 대화가 만들어지는 과정을 처음부터 끝까지 추적한다.

### Step 1: 전략 생성 (H)

```
입력: strategy_prompt.md + 제품 정보 + Strategy Ledger
모델: 비싼 모델 (추론 능력 필요)
출력: 전략 3개 (AIDA 퍼널 구조)
```

### Step 2: 대상 선택 (P)

```
입력: 200명 페르소나
모드: DEV=10명, TEST=50명, DEMO=200명
출력: 이번 실행 대상 페르소나 목록
```

### Step 3: 대화 시뮬레이션 (A)

전략 3개 × 페르소나 200명 = **600개 대화**를 병렬 실행한다.

```
각 대화의 구조:

  세일즈 에이전트 (LLM)
    system prompt = sales-agent-system.md + 이번 전략의 funnel 가이드
    모델: 싼 모델 (대량 호출)

  고객 에이전트 (LLM)
    system prompt = customer-agent-system.md + 해당 페르소나의 soul.md
    모델: 싼 모델

  대화 진행:
    턴 0: 세일즈 → 고객 (Attention → Interest)
    턴 1: 세일즈 → 고객 (Interest → Desire)
    턴 2: 세일즈 → 고객 (Desire → Action)

  동시성: asyncio.Semaphore(20) — 최대 20개 동시 실행
  Retry: exponential backoff (1초 → 2초 → 4초) + task-level 2회 재시도
```

### Step 4: 평가 채점 (E)

600개 대화를 각각 Judge가 채점한다.

```
입력: 대화 transcript
모델: 비싼 모델 (정확한 평가 필요, temperature=0.1)
출력: {engagement, relevance, persuasion, purchase_intent, total, funnel_progress, outcome}
```

### Step 5: 패턴 분석 (R)

```
600개 평가 결과를 점수순 정렬 → 상위 10개 + 하위 10개만 LLM에 전달
→ "뭐가 잘 됐고 뭐가 안 됐는가" 패턴 발견
→ 퍼널 전환 성공률 분석 (A→I: 80%, I→D: 60%, D→A: 15%)
→ 클러스터별 병목 식별
```

**context 폭발 방지**: 600개 대화 원문이 아니라 상위/하위 20개만 분석한다.
나머지 580개는 점수로만 집계된다.

### Step 6: 학습 추출 (L)

```
입력: R의 패턴 분석 + 현재 strategy_prompt.md
출력: 재사용 가능한 학습 포인트 + strategy_ledger.json에 세대 기록 추가
```

## 10. 비즈니스 가치

### 직접적 가치

```
1. 전략 검증 속도
   기존: 전략 1개 검증에 2주 (실제 고객 대상)
   NUDGE: 전략 3개 검증에 5분 (시뮬레이션)
   → 검증 속도 약 600배

2. 고객 이탈 제로
   실패한 전략에 실제 고객이 노출되지 않는다.
   모든 실험은 가상 페르소나에서 진행된다.

3. 노하우 코드화
   베테랑의 감각이 Strategy Ledger에 데이터로 쌓인다.
   proven_effective: 검증된 성공 패턴 → 누구나 재현 가능
   never_repeat: 검증된 실패 → 같은 실수를 반복하지 않음
```

### 간접적 가치

```
4. 클러스터별 맞춤 전략
   "가성비 민감 고객에겐 이 전략, 세일즈 혐오 고객에겐 저 전략"
   → 기존에 불가능했던 고객 유형별 최적 전략 자동 도출

5. 신입 영업사원 온보딩
   10년차의 멘탈맵을 Strategy Ledger로 전달 가능
   "이 유형 고객에겐 이 접근이 3세대 연속 성공했다"

6. 제품 출시 전 전략 사전 검증
   신제품 출시 전에 200명 가상 고객으로 사전 시뮬레이션
   → 어떤 포지셔닝이 먹히는지 미리 파악

7. 지속적 진화
   시장 변화에 따라 페르소나를 업데이트하고 다시 돌리면
   전략이 자동으로 재최적화된다
```

### 핵심 메트릭

```
avg_score (0-100):
  전체 페르소나 평균 설득 점수.
  이 숫자가 올라가면 전략이 진화한 것.

cluster_coverage (%):
  8개 클러스터 중 평균 60점 이상인 비율.
  특정 고객만 잘 설득하는 게 아니라, 전체적으로 잘 먹히는가.

funnel_progress 분포:
  대화들이 퍼널에서 평균 어디까지 도달하는가.
  → 이게 점점 올라가면 "더 깊은 설득"이 일어나고 있다는 뜻.
```

## 11. 기술적 근거 요약

```
이론적 기반:
  • AIDA 모델 (1898, E. St. Elmo Lewis) — 구매 여정 프레임워크
  • Cialdini 설득의 6원칙 — 전략 유형 가이드
  • 진화 알고리즘 — 선택압(evaluation) + 변이(autoresearch) + 유전(ledger)
  • LLM-as-Judge — LLM을 평가자로 사용하는 패턴 (최근 ML 연구)

시스템 설계:
  • 2-레벨 루프: 내부(전략 테스트) + 외부(전략 생성 방법 진화)
  • karpathy/autoresearch 패턴: git keep/discard 기반 메타 최적화
  • Strategy Ledger: 3-레이어 압축으로 context 폭발 방지
  • 200명 심리학 기반 페르소나: 8 클러스터 × 25명, 실제 고객 분포 반영

실행 안정성:
  • Retry: LLM 레이어 exponential backoff + task-level 2회 재시도
  • 동시성: asyncio.Semaphore(20), Gemini RPM 기준 최적화
  • 실패 보고: success_rate 95% 미만 시 자동 재실행
  • 노이즈 관리: Judge temperature=0.1, 200명 평균으로 편차 상쇄
```
